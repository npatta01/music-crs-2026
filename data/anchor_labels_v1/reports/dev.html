Cleaning the training data for a music recommender

How we turn AI-generated chat logs into trustworthy “good / bad recommendation” labels — and what we found on the first 1,000-conversation batch.

The one-sentence version. We're training a system that recommends songs during a chat. Our training data is AI-generated and comes with a built-in “the listener liked it / didn't” flag we don't fully trust — so we have independent AI “judges” re-check every recommendation and produce a clean label (good example / bad example) for the model to learn from.
The specific bug we're chasing — “anchoring.” When a listener says “give me a song by a different artist,” today's recommender often replies with… the same artist again. It ignores the pivot. Example we actually found: the listener said “suggest some different samba artists” and got served another Pixote song — the very artist they wanted to move on from. We're building data that teaches the model not to do that.

Key terms (plain English)

Recommender / retriever — the system that picks which songs to suggest next in the conversation. It's what we're training.
Turn — one step of the chat: the listener says something, the system plays one track. We label each turn.
MOVES / DOES_NOT — the dataset's built-in flag for whether the listener liked (MOVES) or disliked (DOES_NOT) the track. It's AI-generated, so we treat it as a hint, not gospel.
Anchoring — the bug: the listener explicitly asked for a different artist, but the system served the same one.
Judges — two cheap, fast AI models (Gemma, DeepSeek) that read each turn and answer two questions. Arbiter — a stronger, more expensive model (Claude Opus) brought in only when the two judges disagree.
The 5 labels we produce: POSITIVE good example · NEGATIVE bad example (anchoring or wrong genre/era) · DROP ambiguous, set aside · HOLD can't tell, low confidence

324 anchoring negatives = caught instances of the bug. Note many were tagged “liked” in the original data — we override that, because the system still ignored the pivot.

⚠️ This devset is unusually positive (88% liked, few same-artist) because it's an evaluation set. The main training set is ~15× bigger (≈106,000 turns) and far more balanced (~50% liked, ~50% same-artist), so it will surface many more anchoring cases.

When the two judges disagree → Opus reviews it

23% of turns (1,592 here) were disagreements. A stronger model (Claude Opus) re-judges those independently — it answers the same questions fresh, it does not just pick a side. Two real ones:

What we actually ask the AI judges

Each judge gets the last ~3 turns of the conversation (so it can see repeated demands) plus the one track being judged, and answers two questions. Both cheap judges get the identical prompt — only the model differs.

QUESTION 1 — the artist pivot

Did the listener EXPLICITLY ask for a
DIFFERENT / other / new ARTIST (or to
stop a specific one)?  e.g. "other bands",
"someone besides X", "not more X".

FALSE for everything else — "more like
this", a specific song, or a pivot on
genre/mood/era ("no more high-energy
rock"). Judge ONLY the artist.

→ answer: true / false

QUESTION 2 — the content fit

Does the track fit what the listener asked
for in their CURRENT turn? Consider any
named facet: genre, mood, era, tempo, a
specific song...
Handle EXCLUSIONS: if they said "not X /
no more X" and the track still has X → it
violates the ask.

→ answer: valid / invalid / unsure
  (unsure = a subtle feel you can't verify)

How the two answers become a label:   asked-for-different-artist and same artist served → NEGATIVE (anchoring) · content “invalid” → NEGATIVE · content “valid” + liked → POSITIVE · content “valid” + disliked → DROP · content “unsure” → HOLD.  (Whether the artist is the “same” is checked exactly against the music catalog — the AI never guesses that part.)

Technical details

DatasetTalkPlayData (AI-generated conversational music-rec data). Devset = 1,000 conversations / 7,000 labelable turns; training set ≈ 106,000 turns.
Two cheap judgesGemma-4-26B + DeepSeek-V4-Flash (different model families, for diversity). Identical prompt; ~$0.07–0.10 per million tokens.
ArbiterClaude Opus — only on the 23% the judges disagree on. Validated as the trusted reference (93% agreement with hand-checks on the artist question).
Confidence weightsBoth judges agree = 1.0 · Opus-arbitrated = 0.6 · HOLD = 0.3. So the model can down-weight the shaky labels.
Deterministic check“Same artist?” is an exact catalog match in code — not left to the AI.
Quality control2 independent code reviews; error-retry + auto-recovery built in. Devset finished with 0 errors / 0 unresolved.
Cost (devset)≈ $3 (judges) + $0 (arbiter ran on an existing plan). Full training run ≈ $48.
OutputOne clean file per turn: the request, the track, both judges' calls, Opus's call (if used), the final label + reason + confidence weight.

Devset complete — 7,000 turns labeled, 0 errors. Next: the full ~106,000-turn training run, then retrain the recommender and measure whether it stops “anchoring.”
