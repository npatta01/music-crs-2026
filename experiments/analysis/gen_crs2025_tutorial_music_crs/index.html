Start here: what is a CRS?

Recommender system

A system that chooses items for a user. Spotify recommends songs; Netflix recommends movies; Amazon recommends products.

Conversational recommender system

A recommender that works through dialogue. It must remember prior turns, user feedback, and changing preferences.

Generative CRS

A conversational recommender that uses an LLM for some part of the job, such as state tracking, response generation, or reranking.

What the Music-CRS competition asks for

Input
Multi-turn music conversation with user context and prior music recommendations.
->
System task
Understand the current intent using the full conversation, not only the latest turn.
->
Recommendation output
Return 20 valid track IDs from a fixed catalog of about 47,000 tracks.
->
Language output
Generate a natural response grounded in the selected track evidence.
99.6% Conversation-wide dependency

Local query-intent labels show nearly every dev session needs prior user plus assistant/music turns.

0.1072 Offline RRF NDCG@20

Sparse BM25+tags and dense Qwen retrieval already improve when fused.

0.7210 Hybrid Hit@1000

The hybrid candidate pool contains the correct track more often than either branch alone.

The tutorial's architecture lesson

ArchitecturePlain meaningUse for Music-CRS?
Unified Gen-CRS One LLM reads the conversation and directly generates recommendations. Not as the primary path. Too much invalid-ID and popularity-bias risk.
Modular Gen-CRS Separate state tracking, retrieval, reranking, and response generation. Yes. This keeps recommendations grounded in the official catalog.
Agentic Gen-CRS An LLM planner chooses tools and updates memory across steps. Later. Useful but higher latency, harder debugging, and more instability.

Recommended standalone architecture

ConversationState
Structured memory of accepted tracks, rejected tracks, constraints, and current intent.
->
Retrieval views
One sparse keyword query and one dense semantic query derived from the state.
->
Hybrid retrieval
BM25 or LanceDB FTS plus Qwen dense retrieval plus user/item signals.
->
Fusion and rerank
Use RRF first, then feature-aware rescoring over the union pool.
->
Grounded response
Return 20 valid track IDs and generate text only from catalog evidence.

Why ConversationState matters

A flat query can forget that the user rejected a prior suggestion or that "more like this" refers to the second recommendation. A state object keeps those facts separate and reusable.

Why direct LLM recommendation is risky

The LLM can write plausible song names that are outside the catalog. It can also over-recommend popular items. The competition needs valid IDs and measurable ranking quality.

Example ConversationState

intent_mode: refinement
positive_anchors:
  tracks: ["track id from the suggestion the user liked"]
negative_anchors:
  tags: ["too heavy", "too gloomy"]
active_constraints:
  mood: ["chill", "warm"]
  instrumentation: ["more acoustic"]
  vocal: ["female vocalist"]
sparse_query_view: "ambient warm acoustic female vocalist artist"
dense_query_view: "warm chill acoustic electronic song with female vocals"
response_evidence_view:
  allowed_track_facts: ["track name", "artist", "album", "tags"]

Evaluation: what each metric tells you

MetricPlain-English questionWhy it matters
Hit@20 / Recall@20 Did the correct track appear in the final top 20? Measures final recommendation success.
Hit@1000 / Recall@1000 Did candidate generation find the correct track anywhere in the broad pool? A reranker cannot recover a track that was never retrieved.
NDCG@20 Was the correct track ranked near the top? Rewards better ordering, not just inclusion.
Per-turn metrics Does performance break later in the conversation? Late turns expose state and carryover failures.
