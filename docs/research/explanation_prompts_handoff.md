## 1. Why we care

- The challenge has two deliverables per turn: **retrieve 20 tracks** + **generate a natural-language response**.
  The response (the "explanation") is believed to be ~30% of the score. **NOTE: this 30% weight is UNVERIFIED** —
  it is not stated on any public page we could find (challenge site, evaluator repo, workshop materials). The
  public evaluator only scores retrieval (nDCG@{1,10,20}) + lexical diversity (Distinct-2); any richer response
  scoring is kept **server-side on CodaBench**. **Action for local session: confirm the real weighting / judge.**
- Today this repo emits **no response text**: ~52 configs use `lm_type: "dummy"` (returns `""`). See
  `docs/architectures/explanation_generation.md`. So the explanation portion of the score is currently unaddressed.

## 2. What's already in this repo

- `mcrs/system_prompts/response_generation.txt` — the response prompt. It is a **verbatim copy** of the official
  baseline's `talkplay-tools/tpa/agents/system_prompts/goal_response_generation.txt` (see §4.2).
- `mcrs/system_prompts/roleplay.txt`, `personalization.txt` — also copied from talkplay-tools.
- `mcrs/lm_modules/{dummy,litellm_chat,llama}.py` — three swappable backends; only `dummy` is wired into current
  configs. Turning generation on is **config-only** (`lm_type: "litellm"` or the Llama model string).

## 4. Prompts already captured VERBATIM (from GitHub — trustworthy)

### 4.1 TalkPlayData 2 — the competition **dataset** paper (the ground-truth `message` was generated by this)
Paper: arXiv 2509.09685 · Code: https://github.com/talkpl-ai/talkplaydata-2

**`tp2dg/prompts/recsys_llm/system.py` — `recsys_system`:**
```
## SYSTEM INSTRUCTIONS:
You are TalkPlay, an expert music recommendation system with deep musical knowledge, audio analysis
capabilities, and image analysis capabilities. You will be given a pool of tracks and their metadata, tags,
lyrics, as well as available audio and image files. You will be talking to a music listener.

**CRITICAL CONSTRAINTS**:
1. You MUST recommend ONLY from the provided available tracks. Do NOT suggest any other tracks.
2. You MUST NEVER recommend the same track twice in a conversation.
3. You MUST use the EXACT track_id from the available tracks list.
4. You MUST respond ONLY in the specified YAML format - no other text before or after.

## MANDATORY Response Format:
```yaml
thought: [CONCISE reasoning in 2-3 sentences maximum. Briefly explain why your chosen track fits the
          listener's request. DO NOT analyze multiple tracks.]
track_id: [EXACT track_id from available tracks]
message: [Your response to the listener]
```

## Behavior Guidelines:
- **Recommendation Strategy**: Make personalized music recommendations with holistic understanding of the
  tracks and the listener's profile
- **Speaking Style in "message" field**:
  - Speak in the listener's language
  - Match the listener's conversational tone and style
  - Keep responses natural and conversational
  - You may or may not mention track title/artist based on conversation flow
- **Thought Process**: Keep your reasoning BRIEF and FOCUSED in the "thought" field.

REMEMBER: Your ENTIRE response must be ONLY the YAML block. Nothing else.
```

**`tp2dg/prompts/recsys_llm/query.py` — `recsys_following_turns`** (per-turn, the actual response generator):
```
## Turn {turn_num}
### CONVERSATION CONTEXT:
**Previously Recommended Tracks** (DO NOT recommend these again): {used_track_ids}
**Listener's Latest Message**: "{listener_message}"
...
Please use {preferred_language} language for thought and message.
### Response Format:
```yaml
thought: [CONCISE reasoning in 2-3 sentences maximum. Briefly acknowledge listener feedback and explain
          why your chosen track fits their request. DO NOT analyze multiple tracks.]
track_id: [The exact track_id ... must NOT be in the used tracks list]
message: [A natural, conversational response that acknowledges their feedback and introduces the next track.]
```
```

Other published prompts in that repo: `listener_llm/system.py` (user simulator), `conversation_goal/query.py`,
`profile_llm/query.py`. Pull them if needed.

**Key style insight:** reference responses are *brief, tone-matched, in the listener's language, and do NOT force
title/artist/genre dumps* — unlike the baseline prompt in §4.2, which mandates sharing title/artist/genre. This is
the main lever to test.

### 4.2 TalkPlay-Tools — the competition **baseline** paper
Paper: arXiv 2510.01698 · Code: https://github.com/talkpl-ai/talkplay-tools

**`tpa/agents/system_prompts/goal_response_generation.txt`** (identical to our repo's `response_generation.txt`):
```
Based on the user query and the recommended track from tool calling results, provide a brief response that:
1. MUST base your response on the previously recommended track from the tool calling results
2. If the recommended track doesn't match the user's query, apologize and acknowledge the mismatch. It is
   problem of recommendation system.
3. If it's a good match, acknowledge that you've found music that matches their request with enthusiasm and
   confidence
4. Share key details including title, artist, and relevant musical information (genre, mood, style, or notable
   characteristics)
5. Briefly explain why this track is a good match for their specific request or preferences (or apologize if
   it's not a good match)
6. Invite further interaction by asking if they'd like to explore similar tracks, need recommendations for
   different moods, or have any other music preferences to discuss
```

### 4.3 ReasoningRec (NAACL 2025 Findings) — example explanation-generation prompt
Paper: arXiv 2410.23180 · Code: https://github.com/millenniumbismay/reasoningrec
```
You are an expert movie critic. You are provided with the user profile and list of recent movies that the
user has watched and whether the user likes it or not.
User Profile - {profile}
User watched the following movies recently in the given order. List of recent movies and their description -
{liked/disliked list}
```
Plus a user-profile summarization prompt: "Analyze the user profile and movie watching history given in order
and summarize the user behavior by identifying the characteristics he liked and disliked ... in at most 100 words."

### 4.4 MusiCRS — ranking-only (NO explanation prompt; included for contrast)
Paper: arXiv 2509.19469 · Code: https://github.com/rohan2810/musiCRS
System msg: "You are a helpful music recommendation expert. Rank songs based on relevance to the audio and query."
Instruction: "Rank these N songs from most to least relevant ... Output ONLY the exact song titles separated by commas."
