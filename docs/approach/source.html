01 · The 60-second lifecycle

One conversation, nine explicit handoffs

The system turns fuzzy language into a traceable recommendation through small contracts. Every stage leaves evidence that the next stage can consume—and that we can later inspect.

  1. Hear the conversation

    Read the current request in the context of the listener’s earlier turns.

    Input schema

  2. Extract typed state

    Translate natural language into intent, anchors, preferences, constraints, and routing tags.

    Active config

  3. Resolve catalog entities

    Match mentioned tracks and artists to catalog identities while preserving unresolved descriptive clues.

    Saved trace

  4. Compile retrieval work

    Turn the resolved state into branch-specific lexical, semantic, audio, visual, and lookup queries.

    Compiler code

  5. Retrieve candidates

    Let each eligible specialist search deeply, producing ranked candidate pools rather than a premature answer.

    Saved trace

  6. Fuse branch signals

    Combine heterogeneous ranked lists with weighted reciprocal-rank fusion into a shared candidate union.

    Verified mechanics

  7. Rerank the shortlist

    Apply the goal-free LightGBM bundle and final guards, then select the twenty tracks to return.

    Active model

  8. Explain the first choice

    Condition a concise natural-language reply on the latest state and the primary recommended track.

    Submission config

  9. Persist and evaluate

    Write predictions and traces; where labels exist, score ranking quality and diagnose stage boundaries.

    Labelled devset

02 · Workload placement

Local control, selective remote compute

“Local” describes the runner, not necessarily every model call. The active online configuration keeps orchestration and catalog search local, calls hosted APIs for extraction and Qwen embeddings on cache misses, and uses Modal for CLAP and SigLIP2 text encoders.

  1. Local

    Orchestration, caches, and catalog

    The Python runner loads YAML, checks file-per-turn state and embedding caches, resolves entities, and searches the local LanceDB catalog. The active staged devset pipeline also shards retrieval locally.

  2. Hosted API

    Language and Qwen embedding calls

    On a genuine cache miss, DeepSeek V4 Flash extracts state through LiteLLM; Qwen3 0.6B and 8B query embeddings use configured OpenAI-compatible endpoints.

    Modal

    Audio and cover-art query encoders

    Eligible sonic and visual routes invoke the deployed MultimodalTextEncoder methods for CLAP and SigLIP2 text embeddings. Modal can alternatively host an entire retrieval run when the runner backend is switched.

  3. Persisted artifact

    Predictions, traces, and a manifest

    Run-scoped JSON, JSONL, logs, scores, and manifest metadata make the expensive retrieval result reusable instead of ephemeral.

  4. Local

    Replay and evaluation

    The staged runner replays the committed LightGBM bundle over a saved trace, merges shards, writes the top 20, and evaluates devset output locally. Staged explanation replay is intentionally dummy-only.

Boundary check: the canonical staged devset YAML currently sets both retrieval and rerank to local. A mixed run may set only retrieval to modal; LightGBM replay remains local. Hosted API and Modal calls inside an online retrieval configuration are separate from the runner’s backend label.

Configuration map: what controls placement?

configs/pipelines/state_ranker_v10_lgbm_devset.yaml sets the staged runner, retrieval, and rerank backends to local and points replay at models/reranker_v12_goalfree.

configs/state_ranker_v10_lgbm_devset.yaml sets the extractor and Qwen encoder endpoints, Modal multimodal methods, LanceDB path, candidate depths, fusion, and final LightGBM paths.

run_pipeline.py enforces local-only rerank replay, persists run roots under exp/pipeline/runs, and invokes the devset evaluator after merge.

Zero-cost offline reproduction

The offline bundle can replay the shipped splits with credentials blanked and the network fenced. Its activation script points state, embedding, and LiteLLM lookups at curated caches and enables MCRS_REQUIRE_LITELLM_CACHE=1, so a miss fails instead of spending money.

Frozen predictions can be byte-verified. A fresh offline ANN run can reproduce comparable recommendations but is not promised to be byte-identical because approximate-nearest-neighbor ordering can vary; blind-set relevance still cannot be scored locally because organizer labels are hidden.

04 · State extraction

Preserve what matters for the next recommendation

The production extractor is deliberately not a transcript summarizer. It asks for a narrow, evidence-linked V1 object, then a deterministic bridge derives the compatibility fields the existing resolver and compiler expect.

Production prompt

A bounded extraction contract

Next-recommendation scope: “Extract state for the NEXT recommendation at the latest user turn.” The prompt classifies requests such as exact track, same artist, new artist, similar-to-prior, attribute search, and hidden target. Facts carry a role—current target, satisfied prior, history, contrast, or rejected—and an anchor_use that controls whether they seed retrieval. Explicit future exclusions are separate from descriptive negatives.
Compatibility boundary: the LLM must not emit turn_intent, entities, rejections, target_artist_mode, or retrieval_profile; the deterministic bridge derives them from the V1 fields.

Meaningful excerpts condensed from the active state prompt; wording is bounded to the rules needed to understand this turn.

05 · Resolution and retrieval

Specialists search independently, then meet again

Resolution first separates catalog identities from descriptive language. The compiler then fans one resolved state into specialist queries. A skipped branch is an explicit routing outcome, not missing telemetry.

Resolver output

Surface names → catalog IDs

This turn: there were no artist, album, or track surface names, so the resolver returned no anchor or exact-target IDs. It preserved all three visual phrases as positive tags. In other turns, named surfaces are matched to stable catalog IDs before discography, centroid, protection, or exclusion logic runs.

↓ reconverge
Saved candidate union

One union for one ranker

The five fired lists contained up to 5,000 branch positions. Deduplication produced 4,640 unique candidates; all 4,640 remained eligible because no release-date, played-track, rejection, or artist-policy filter dropped an item. Weighted reciprocal-rank fusion then produced one 1,000-item ordered candidate stage for learned reranking.

Structural invariant: the fan-out ends here. Every retrieval path—whether lexical, semantic, audio, image, centroid, or lookup—must reconverge before the final ranker and response handoff.

Exact trace status for all configured branches

Fired: bm25, dense.qwen_8b.metadata, dense.qwen_8b.attributes, dense.clap_text.sonic_nl, dense.siglip2_text.visual_nl—1,000 raw hits each. Skipped: lyrics (no_query); three anchor centroids (no_anchors); user CF (missing_user_vector); discography and era/popularity (no_query).

06 · Ranking

Retrieval sets the ceiling; the judge changes the order

A ranker can promote only candidates that retrieval supplied. Here the candidate stage is weighted RRF; the final stage is the models/reranker_v12_goalfree LambdaMART bundle, recorded under the stable trace label lgbm_v10.

Verified ranking mechanics

What the learned judge can see

Feature families include branch rank/score/margin signals; catalog priors such as popularity, era, duration, and tag coverage; request type and routing flags; constraints and exact-match tiers; session/history relationships; lexical and dense query-to-candidate similarities; collaborative, audio, and image affinities; and guard features for played or rejected items.

b1_cos adds a goal-free conversation-to-track score: the current conversation and each candidate’s precomputed track representation live in the same fine-tuned Qwen3-Embedding-4B space, and their cosine similarity becomes one LightGBM feature. In the submitted scout-only setup, it judges candidates already retrieved; it is not a separate candidate branch.

See b1 bi-encoder architecture and the served model’s feature manifest.

07 · Response handoff

The language model presents one already-selected track

Generation does not choose among the 20. The submitted path passes only rank one to Qwen3-30B-A3B, alongside latest-state conditioning, and asks for a short listener-facing explanation.

Submitted response context

What crosses the handoff

  • Latest conversation stateThe latest user request, the derived current request, and the three positive mentioned entities rendered as Liked / wants. Optional Disliked / avoid, explicit-rejection, year-range, and lyrical-theme lines are appended only when populated; none were rendered for this turn.
  • Selected item blockThe rank-one track’s title “Pink + White,” artist Frank Ocean, album Blonde, and at most ten catalog tags, rendered as delimited XML.
  • Style contractOne or two concise sentences, prioritize the latest ask, explain one supported reason, and avoid overselling a clear mismatch.
Production response prompt

Explain; do not select

“A separate recommendation system has ALREADY selected one track to play next.” The model’s job is to acknowledge the request and explain why that track fits. It must use only supported facts, avoid dumping tags or XML, match the listener’s language and tone, and respond in 1–3 sentences.

The final listener-facing prose

The cover of Frank Ocean’s Blonde is a striking, abstract piece—soft pink and white gradients layered with blurred, painterly shapes that feel like a modern art canvas. It matches your vibe perfectly, with its bold yet dreamy visual style that’s as evocative as the music.

This text is reproduced verbatim from the saved prediction artifact. It proves that generation ran; it does not prove the track or description was relevant or factually correct.

Grounding boundary

Where the evidence ends

The item context contains only title, artist, album, and capped tags—not an independently verified prose description of the cover. Although the prompt says to invent nothing, the saved response’s claims about gradients, blurred shapes, and a “perfect” match are not established by the visible trace or catalog row. They therefore remain unverified generation claims.

Submitted control path: top-1, single-pass generation. We found no independently documented fact checker and no selector/critic that compares alternative responses or chooses another track after prose generation. The generator explains the ranker’s first choice; it does not audit or revise that choice.

Response settings and audit sources

The Blind-B config selects phase2_best_qwen, conditioning: latest_state, item_format: xml, max_tags: 10, temperature: 0.0, and Qwen3-30B-A3B via LiteLLM. The earlier small bake-off supports the configuration choice but is not a Blind-B quality score.
