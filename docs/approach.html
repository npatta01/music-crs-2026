03 · State extraction

Preserve what matters for the next recommendation

The production extractor is deliberately not a transcript summarizer. It asks for a narrow, evidence-linked V1 object, then a deterministic bridge derives the compatibility fields the existing resolver and compiler expect.

Production prompt

A bounded extraction contract

Next-recommendation scope: “Extract state for the NEXT recommendation at the latest user turn.” The prompt classifies requests such as exact track, same artist, new artist, similar-to-prior, attribute search, and hidden target. Facts carry a role—current target, satisfied prior, history, contrast, or rejected—and an anchor_use that controls whether they seed retrieval. Explicit future exclusions are separate from descriptive negatives.
Compatibility boundary: the LLM must not emit turn_intent, entities, rejections, target_artist_mode, or retrieval_profile; the deterministic bridge derives them from the V1 fields.

04 · Resolution and retrieval

Specialists search independently, then meet again

Resolution first separates catalog identities from descriptive language. The compiler then fans one resolved state into specialist queries. A skipped branch is an explicit routing outcome, not missing telemetry.

Resolver output

Surface names → catalog IDs

Saved candidate union

One union for one ranker

Seven 1,000-hit retrieval lists plus the 200-hit era lookup deduplicated to 5,970 unique candidates. Structural invariant: the fan-out ends here. Every retrieval path—whether lexical, semantic, audio, image, centroid, or lookup—must reconverge before the final ranker and response handoff.

How candidate pool assembly combines branch evidence

The compiler uses internal weighted reciprocal-rank fusion: score(track) = Σ branch_weight / (60 + rank_in_branch). This rank-based score combines otherwise incomparable branch scales and orders the candidate stage supplied to LightGBM; it is not the submitted final ranker.

Exact trace status for all configured branches

Each centroid records source IDs for “Pumped Up Kicks,” “Sit Next to Me,” and “Houdini.” Era/popularity fired with 200. Skipped: lyrics (no_query), SigLIP2 text (gated_off), user CF (missing_user_vector), and resolved-artist discography (no_query; resolved artist present but role-ineligible).

05 · Ranking

Retrieval sets the ceiling; the learned ranker changes the order

ranking.mode: lgbm is the submitted final path. The compiler’s candidate pool assembly first combines the specialist branches. The goal-free LambdaMART bundle runs next: LightGBM determines the delivered order, followed by the same-turn artist guard.

Verified ranking mechanics

What the learned ranker can see

Feature families include branch rank/score/margin signals; catalog priors such as popularity, era, duration, and tag coverage; request type and routing flags; constraints and exact-match tiers; session/history relationships; lexical and dense query-to-candidate similarities; collaborative, audio, and image affinities; and guard features for played or rejected items.

b1_cos adds a goal-free conversation-to-track score: the current conversation and each candidate’s precomputed track representation live in the same fine-tuned Qwen3-Embedding-4B space, and their cosine similarity becomes one LightGBM feature. In the submitted scout-only setup, it judges candidates already retrieved; it is not a separate candidate branch.

See b1 bi-encoder architecture and the served model’s feature manifest.

06 · Response handoff

The language model presents one already-selected track

Generation does not choose among the 20. The submitted path passes only rank one to Qwen3-30B-A3B, alongside latest-state conditioning, and asks for a short listener-facing explanation.

07 · Workload placement

Local control, selective remote compute

“Local” describes the runner, not every model call. The competition workflow lacked a suitable local discrete GPU, so targeted remote GPU capacity handled the model-specific work below while orchestration, caches, catalog search, reranking, and evaluation stayed local.

  1. Local

    Orchestration, caches, and catalog

    The Python runner loads YAML, checks file-per-turn state and embedding caches, resolves entities, and searches the local LanceDB catalog. The active staged devset pipeline also shards retrieval locally.

  2. Hosted API · OpenRouter

    DeepSeek state extraction

    On a genuine state-cache miss, LiteLLM routes deepseek-v4-flash through OpenRouter to produce the V1 conversation state.

    Hosted API · DeepInfra

    Qwen3-Embedding-0.6B

    Lyrics queries and tag-resolution embeddings use the config’s OpenAI-compatible DeepInfra endpoint, behind the shared disk cache.

    Modal · vLLM

    Qwen3-Embedding-8B

    The logical qwen3-embedding-8b endpoint resolves to the private Modal vLLM service for metadata and attribute query embeddings.

    Modal

    CLAP and SigLIP2

    Sonic and cover-art routes invoke the private MultimodalTextEncoder methods for CLAP-text and SigLIP2-text embeddings.

    Hosted API · OpenRouter

    Qwen response generation

    After ranking, LiteLLM sends the top-one explanation handoff to qwen3-30b-a3b-instruct-2507 through OpenRouter.

  3. Persisted artifact

    Predictions, traces, and a manifest

    Run-scoped JSON, JSONL, logs, scores, and manifest metadata make the expensive retrieval result reusable instead of ephemeral.

  4. Local

    Replay and evaluation

    The staged runner replays the committed LightGBM bundle over a saved trace, merges shards, writes the top 20, and evaluates devset output locally. Staged explanation replay is intentionally dummy-only.

Boundary check: the submitted Blind-B YAML and endpoint-resolution code establish this placement. A cache hit can avoid a remote call, and a runner labelled local can still invoke these private hosted services; the runner backend and individual model placement are separate controls.

Configuration map: what controls placement?

configs/pipelines/state_ranker_v10_lgbm_devset.yaml sets the staged runner, retrieval, and rerank backends to local and points replay at models/reranker_v12_goalfree.

configs/state_ranker_v10_lgbm_devset.yaml sets the extractor and Qwen encoder endpoints, Modal multimodal methods, LanceDB path, candidate depths, fusion, and final LightGBM paths.

run_pipeline.py enforces local-only rerank replay, persists run roots under exp/pipeline/runs, and invokes the devset evaluator after merge.

Zero-cost offline reproduction

The offline bundle can replay the shipped splits with credentials blanked and the network fenced. Its activation script points state, embedding, and LiteLLM lookups at curated caches and enables MCRS_REQUIRE_LITELLM_CACHE=1, so a miss fails instead of spending money.

Frozen predictions can be byte-verified. A fresh offline ANN run can reproduce comparable recommendations but is not promised to be byte-identical because approximate-nearest-neighbor ordering can vary; blind-set relevance still cannot be scored locally because organizer labels are hidden.

08 · Challenge evaluation

Four signals become one score

The official composite rewards retrieval relevance most, but it also rewards breadth and the quality of the listener-facing reply. Each term is normalized to a zero-to-one scale before weighting.

0.50 × nDCG@20 + 0.10 × catalog diversity + 0.10 × lexical diversity + 0.30 × (response judge − 1) / 4
  • nDCG@20 · 50%Measures whether relevant tracks appear near the top of the delivered twenty, discounting useful items more heavily as they move down the list. In the project’s single-target devset diagnostics, a missing target has zero gain and a top-ranked target has the strongest gain.
  • Catalog diversity · 10%Measures catalog coverage: the number of unique tracks recommended across turns divided by the 47,071-track catalog. It asks whether the system uses more than a narrow, repetitive slice.
  • Lexical diversity · 10% · official Distinct-2The challenge term measures variation in generated response language through distinct bigrams. The local evaluator additionally reports Distinct-1 as a diagnostic alongside Distinct-2; retrieval-only dummy responses score zero by construction.
  • Normalized response judge · 30%The challenge LLM judge assigns the listener-facing reply a score from 1 to 5. The formula maps that value with (judge − 1) / 4, so 1 becomes 0 and 5 becomes 1 before applying the 30% weight.

Ranking relevance

Recall and nDCG evaluate whether relevant tracks are retrieved and ordered.

Response quality

LLM-as-judge evaluates listener-facing prose offline; it does not rank tracks in the serving pipeline.

Official and development-time language-model judgments

The organizer-side challenge judge evaluates the submitted natural-language response and directly contributes 30% of the composite. Its prompt is undisclosed, and a saved response or local trace does not reveal this score.

Separate proxy judges and LLM-generated relevance labels helped compare response variants or build training evidence. They are development tools, not the challenge judge; the eight-session English proxy was explicitly treated as a coarse analysis signal rather than an official outcome measure.

Scope boundary: labelled devset traces below support success and failure claims. Blind-B traces show the final system’s mechanics, while organizer scores summarize the submission as a whole; hidden per-turn Blind-B relevance prevents attributing its result to any architecture difference or declaring an individual Blind-B example good or bad.

Glossary of retrieval, ranking, and evaluation terms
BM25
A lexical ranking method that scores documents from query-term matches while balancing term rarity and document length.
ANN
Approximate nearest-neighbor search: a fast way to retrieve vectors close to a query vector without exhaustively comparing the entire catalog.
Centroid
The average vector of a set of embeddings, used here as a compact representation of a group of tracks or preferences.
RRF
Reciprocal-rank fusion: a method that combines ranked lists by awarding each candidate a diminishing score based on its position in each list.
LambdaMART
A learning-to-rank method built from boosted decision trees and trained to improve ordering metrics; LightGBM implements the final ranker used here.
Bi-encoder
A model that encodes the conversation and track separately into the same vector space, allowing their similarity to be computed efficiently.
nDCG@20
Normalized discounted cumulative gain over the first 20 results: a ranking metric that rewards relevant items more when they appear nearer the top.
LLM judge
A language model used to assign a quality score to generated text. The organizer’s official response judge is distinct from development-time proxy judges.
Grounding
Keeping a generated claim tied to evidence supplied by the conversation, catalog, or trace rather than unsupported model knowledge or invention.
Candidate recall
The share of relevant targets present in the candidate pool before final reranking; a missing candidate cannot be recovered by the ranker.
Evidence-status boundary
Verified
Directly supported by an inspected repository source, saved artifact, labelled trace, or recorded configuration. Verification applies only to the claim and scope named.
Inferred
A reasoned interpretation that follows from verified evidence but is not itself recorded directly; it must not be presented as an observed causal result.
Illustrative
An example used to explain mechanics or show an output when relevance or outcome labels are unavailable; it does not establish quality or success.

Evidence rule: source code verifies intended mechanics, saved traces verify recorded execution, and labelled evaluation artifacts verify outcomes within their stated split. None alone proves that a component caused a score change. Blind-B examples remain illustrative where per-turn labels are hidden.
